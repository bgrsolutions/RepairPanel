# Email Transport Configuration

Phase 17.3 adds production-ready email delivery to IRONCore RepairPanel.

## Transport Modes

### Log Mode (Default)

In log mode, emails are logged to the application logger but not delivered. This is the default for development and testing.

```bash
MAIL_TRANSPORT=log  # or simply omit the variable
```

### SMTP Mode

In SMTP mode, emails are delivered via an SMTP server.

```bash
MAIL_TRANSPORT=smtp
```

## Environment Variables

| Variable | Default | Description |
|----------|---------|-------------|
| `MAIL_TRANSPORT` | `log` | Transport mode: `log` or `smtp` |
| `MAIL_SMTP_HOST` | `localhost` | SMTP server hostname |
| `MAIL_SMTP_PORT` | `587` | SMTP server port |
| `MAIL_SMTP_USERNAME` | (empty) | SMTP authentication username |
| `MAIL_SMTP_PASSWORD` | (empty) | SMTP authentication password |
| `MAIL_SMTP_USE_TLS` | `true` | Use STARTTLS (port 587) |
| `MAIL_SMTP_USE_SSL` | `false` | Use SSL/TLS (port 465) |
| `MAIL_SMTP_TIMEOUT` | `10` | Connection timeout in seconds |
| `MAIL_DEFAULT_SENDER_EMAIL` | (empty) | From email address |
| `MAIL_DEFAULT_SENDER_NAME` | (empty) | From display name |
| `MAIL_DEFAULT_REPLY_TO` | (empty) | Reply-To email address |

### Sender Fallback Priority

1. `MAIL_DEFAULT_SENDER_EMAIL` / `MAIL_DEFAULT_SENDER_NAME` environment variables
2. Active company record email / display name
3. `noreply@example.com` / `IRONCore RepairPanel` (fallback defaults)

## SMTP Examples

### Gmail / Google Workspace

```bash
MAIL_TRANSPORT=smtp
MAIL_SMTP_HOST=smtp.gmail.com
MAIL_SMTP_PORT=587
MAIL_SMTP_USERNAME=your-email@gmail.com
MAIL_SMTP_PASSWORD=your-app-password
MAIL_SMTP_USE_TLS=true
MAIL_DEFAULT_SENDER_EMAIL=your-email@gmail.com
MAIL_DEFAULT_SENDER_NAME=Your Repair Shop
```

### Generic SMTP (port 465 SSL)

```bash
MAIL_TRANSPORT=smtp
MAIL_SMTP_HOST=mail.yourdomain.com
MAIL_SMTP_PORT=465
MAIL_SMTP_USERNAME=noreply@yourdomain.com
MAIL_SMTP_PASSWORD=your-password
MAIL_SMTP_USE_TLS=false
MAIL_SMTP_USE_SSL=true
MAIL_DEFAULT_SENDER_EMAIL=noreply@yourdomain.com
MAIL_DEFAULT_SENDER_NAME=Your Repair Shop
MAIL_DEFAULT_REPLY_TO=support@yourdomain.com
```

## Sending a Test Email

1. Log in as an Admin user
2. Navigate to **Settings** → **Email Settings**
3. Enter a recipient email address in the "Send Test Email" section
4. Click **Send Test Email**
5. Success or failure is shown as a flash message

The test email uses the current transport configuration. In log mode, it will "succeed" (logged only). In SMTP mode, it will attempt real delivery.

## Email Types

The branded email system supports these email types:

| Type | Template | Trigger |
|------|----------|---------|
| Warranty Confirmation | `warranty_confirmation.html` | Warranty created on ticket |
| Warranty Expiry Reminder | `warranty_expiry_reminder.html` | Manual or scheduled |
| Aftercare Follow-up | `aftercare_followup.html` | Manual from ticket detail |
| Ticket Update | `ticket_update.html` | Staff sends customer update |
| Test Email | `test_email.html` | Admin test from settings |

All templates are available in English and Spanish (`emails/en/` and `emails/es/`).

## Error Handling

- SMTP connection failures, authentication errors, and delivery errors are caught gracefully
- Failed sends never crash business workflows (tickets, warranties, etc.)
- All delivery attempts are logged with status, transport, recipient, and error details
- The `EmailResult` object returned by all send functions includes `success`, `message`, `error`, and `transport` fields

## Delivery Logging

Every email send attempt produces a structured log line:

```
EMAIL_DELIVERY: status=success transport=smtp to=customer@example.com subject=... template=... event=warranty_confirmation error=-
```

This enables monitoring and debugging of email delivery in production logs.

## Permissions

- Email settings page: requires `can_manage_settings` (Admin only)
- Test email action: requires `can_manage_settings` (Admin only)
- Sending branded emails: requires `can_send_branded_email` (Management, Front Desk, Workshop)

## Architecture Notes

- Transport is Python-native using `smtplib` — no external mail libraries required
- HTML emails include a plain-text fallback (auto-generated by stripping HTML tags)
- All emails use the branded base template (`emails/base_email.html`) with company header/footer
- Configuration is environment-variable-driven — no secrets in code
- Switching between log and SMTP mode requires only changing `MAIL_TRANSPORT`
